架构总结

2021-10-08

一、访问远程服务

1. 进程间通信(IPC)

  • 管道、具名管道

    管道允许在进程间传递少量的字符流和字节流。普通管道只用于有亲缘关系的进程间通信，具名管道因为摆脱了普通管道没有名字的限制，允许无亲缘关系的进程间通信。其常见的应用就是命令行中的 | 操作

    将 ps 进程的输出连接到 grep 进程上

  • 信号

    异步通信机制，一个进程不必通过特定操作等待信号的到达，进程也不知道信号什么时候到达，但只能承载少量信息，

    shell 进程向 Pid 进程发送 kill 信号

  • 信号量

    相当于在操作系统定义一个全局变量，会在创建时赋予初始值，初始值为1就是互斥锁，一个任务想要访问共享资源，就必须先获得信号量，获得信号量后值会-1，释放后恢复。负几就表示有几个进程正在等待访问共享资源

  • 消息队列

    上述三种方式只能用于传输少量信息，消息队列克服了信号承载信息量少，管道只能用于无格式字节流以及缓冲区大小受限等缺点，但实时性相对受限

  • 共享内存

    允许多个进程访问同一块公共的内存空间，通常和其他通信方式结合，如：信号量，来达到进程间同步及异步的协调操作

  • 套接字接口

    以上方式只适用于单机多进程通信，套接字可以在不同机器的进程间通信，针对本机的进程间通信会优化，不需要经过网络协议栈，所以断网情况下也能ping本地

2. 远程进程通信(RPC)

由于远程通信的成本问题，如：网络受延迟、带宽、安全性限制；以及被调用方异常的处理、请求超时的处理等需要针对特定业务场景的处理。rpc的通信细节不适合隐藏在操作系统底层，而应该是高层次的或者说语言层次的特征

不同的 rpc 协议主要通过各自的手段解决以下三个基本问题

  • 如何序列化/反序列化数据

    • ONC RPC 的External Data Representation （XDR）
    • CORBA 的Common Data Representation（CDR）
    • Java RMI 的Java Object Serialization Stream Protocol
    • gRPC 的Protocol Buffers
    • Web Service 的XML Serialization
    • 众多轻量级 RPC 支持的JSON Serialization

  • 如何传输数据

    这里的传输指应用层协议，实际传输一般基于标准的tcp、udp等完成

    • Java RMI 的Java Remote Message Protocol（JRMP，也支持RMI-IIOP）
    • CORBA 的Internet Inter ORB Protocol（IIOP，是 GIOP 协议在 IP 协议上的实现版本）
    • DDS 的Real Time Publish Subscribe Protocol（RTPS）
    • Web Service 的Simple Object Access Protocol（SOAP）
    • 如果要求足够简单，双方都是 HTTP Endpoint，直接使用 HTTP 协议也是可以的（如 JSON-RPC）

  • 如何确定方法

    • Android 的Android Interface Definition Language（AIDL）
    • CORBA 的OMG Interface Definition Language（OMG IDL）
    • Web Service 的Web Service Description Language（WSDL）
    • JSON-RPC 的JSON Web Service Protocol（JSON-WSP）

3. 表征状态转移(REST)

表征 ：即资源的表现形式或者叫信息与用户交互的形式，可能是html、markdown、pdf、rss等

状态 ：服务器是否维护上一请求与下一个请求间的上下文，如：分页时，用户当前阅读页维护在客户端则为无状态；维护在服务端则为有状态。个人理解为同一资源在不同客户端需求下展现的不同状态

转移：不论是客户端对表征还是状态的变化都需要通过请求服务端的方式来实现，比如：客户端访问电子书请求下一页请求，服务端将 ‘电子书’ 这个资源进行页码状态变化后以application/json的表征形式提供给客户端

RESTful

rest 和 restful 的关系是，rest的作用范围是架构风格与网络的软件架构设计，而restful是远程服务调用的设计，前者是后者的超集，而restful的系统需要满足以下六个原则：

  • 服务端和客户端分离
  • 无状态
  • 可缓存：因为无状态的设计导致每次请求的上下文都维护在客户端，原来一个请求就能完成的业务可能需要请求多次，或者在请求中带有额外的冗余信息，所以允许客户端和中间的通讯传递者将部分的服务端应答缓存起来，以进一步提高性能
  • 分层系统：客户端不许知道请求经过了哪些中间服务器到达最终的服务器，中间服务器可以实现内容分发网络、负载均衡、共享缓存能功能
  • 统一接口：即面向资源编程，所有针对资源的操作通过http协议中固有的操作命令完成
  • 按需代码：主要应用于前端，逻辑代码存储在服务端，只有当被请求后才会将代码传输到客户端执行，执行后销毁

REST的接口设计相比rpc主要有以下优势

  1. 语义类似，降低的服务接口的学习成本
  2. 因为资源天生具有层次结构和集合，可以很方便的根据资源的上下级定义/来间隔
  3. rest复用HTTP 协议中已经定义的概念和相关基础支持来解决问题，而http协议已经无比成熟

同时也存在以下劣势

  1. 面向资源的编程思想只适合做 CRUD，面向过程、面向对象编程才能处理真正复杂的业务逻辑
  2. 与http绑定，不适用于高性能传输场景
  3. 相比于 WS-AtomicTransaction、WS-Coordination 这样的天生支持事务的功能性协议，http显然是不支持分布式事务的，不过这不止是rest的问题
  4. http没有传输可靠性支持，意味着没有收到响应可能是发送时出现问题也可能是响应时出现问题，所以就要求接口幂等，http协议要求GET、PUT 和 DELETE 应具有幂等性，我们把 REST 服务映射到这些方法时，也应当保证幂等性；针对post浏览器一般会做一些预处理，如：是否重复提交表单的确认
  5. 针对多个资源单位或者获取资源部分字段的请求，处理能力不够强。有针对这方面的中间框架graphql，但因为不好在数据库层面性能优化、接口鉴权等问题并没有推广开来

二、事务处理

1. 本地事务

事务主要就是为了保证数据状态的一致性(Consistency)，这需要以下三个方面的努力(AID)

  • 原子性：在一个事务范围内，对数据的修改，要么全部成功，要么全部失败
  • 隔离性：在不同的事务范围内，对正在读、写的数据互相独立
  • 持久性：保证已经被成功提交的数据都能被正确持久化

原子性和持久性

实现 原子性 最大的困难是写入操作不是原子的，有写入、未写入和正在写状态，所以可能出现以下两种情况

  1. 未提交事务，写入后崩溃：重启后数据库需要得知crash前发生的不完整操作，并将数据恢复为没有改动过的样子，以保证原子性
  2. 已提交事务，写入前崩溃：重启后数据库需要得知crash前发生的不完整操作，并将还未来得及写入磁盘的数据重新写入，以保证持久性

由于写入的中间状态和crash都不可避免，所以实现的重点要放在崩溃恢复上，而崩溃恢复的方法主要有以下几种形式：

  • Commit Logging

    每次执行数据库修改操作，将修改的数据原始值、变化后的值、磁盘位置以顺序追加的方式将日志记录到磁盘上，同时也会记录事务提交消息和结束提交消息用于查验是否数据真的被持久化成功,这个日志被称为 redo log-重做日志(可以解决已提交事务，写入前崩溃)

    这种方式的缺点是只能在发生事务提交后才真正开始执行磁盘io操作，无论磁盘io多么空间、事务修改的数据量多么大、占用了多少内存缓冲区，都不允许在提交事务前执行io，所以对提升数据库性能十分不利

  • Write-Ahead Logging

    允许在提交事务前就修改数据，将已经修改了哪些值，磁盘位置等记录为日志，以便发生崩溃或者回滚时根据Undo Log对提前写入的数据进行擦除，这种日志被称为 undo log-回滚日志(可以解决未提交事务，写入后崩溃)

隔离性

实现隔离性，现代数据库主要提供了以下三种锁

  • 写锁(X-Lock) ：如果数据有加写锁，就只有持有写锁的事务才能对数据进行写入操作，数据加持着写锁时，其他事务不能写入数据，也不能施加读锁
  • 读锁(S-Lock)：多个事务可以对同一个数据添加多个读锁，数据被加上读锁后就不能再被加上写锁，所以其他事务不能对该数据进行写入，但仍然可以读取。对于持有读锁的事务，如果该数据只有它自己一个事务加了读锁，允许直接将其升级为写锁，然后写入数据
  • 范围锁：对某个数据范围加排他锁，做个排他锁的作用范围不止是当前范围内的数据，也不能在这个范围内新增或删除任何数据

mysql InnoDB的行锁是针对索引加的锁，不是针对记录加的锁。并且该索引不能失效，否则都会从行锁升级为表锁。不同的数据库隔离级别通过使用不同的锁在不同加锁时间组合来实现：

  • 可串行化：同时使用了写锁、读锁和范围锁

  • 可重复读：使用写锁和读锁，直到事务结束，会产生幻读问题，即一个事务范围内对两个完全相同的范围查询结果不同

    ps:mysql可重复读隔离级别有优化，只读事务没有幻读问题，读写事务依然存在幻读

  • 读已提交：使用写锁到事务结束，读锁在读取完成后马上释放，会产生不可重复读问题，即在一个事务范围内对同一条数据查到不同的结果

  • 读未提交：只对修改的数据加写锁，可能出现脏读问题，即在事务内读取另一个事务回滚的数据

除了通过锁的方式实现不同的隔离级别外，近年来还有一种通过版本控制实现隔离的方式，即 MVCC 多版本并发控制。MVCC的基本思想是通过版本戳下新老数据共存来实现读取不需要加锁的优化策略，给每个事物一个自增的id，然后如果隔离级别是可重复读，就获取版本小于当前事务id的记录；如果是隔离级别是读已提交，就直接获取最新的版本的记录

2. 分布式事务

2.1 XA协议、2PC、3PC

X/Open组织在1991年提出了一套叫 XA 的处理事务框架，用于DTP(分布式事务处理)，目前可以作用于同一服务下多个数据源的场景。

其核心是定义了一组分布式协调组件间的通信接口，组件间通过通信来协调多个数据源的一致操作，实现统一提交或者统一回滚，在java中也有针对XA协议的实现——JTA，XA协议组件主要包括以下：

  • 事务管理器 如果是应用程序访问一堆数据库，那它是那个应用程序。如果是一群数据库之间（譬如你直接发送SQL语句）的XA事务，那它是发起事务的那个数据库。
  • 资源管理器 一般指数据库实例

基于XA接口规范衍生出了二段提交(2PC)和三段提交(3PC)协议

二段提交

  • 准备阶段：协调者询问所有的参与者是否准备好提交（记录完成redo log，但没有记录’Commit Record’）
  • 提交阶段：如果所有参与者都返回了准备完成，则协调者先提交本地事务(记录’commit record’)，然后通知所有参与者提交本地事务；如果有一个参与者返回失败，则贴条这先将自己记录根据undo log回滚和通知所有参与者回滚本地事务

在这个过程中存在一下问题：

  • 单点问题：协调者宕机，参与者一直没收到提交或回滚消息，导致资源一直被阻塞无法使用
  • 性能问题：整个上锁过程持续到参与者中最慢的一个完成处理操作，并且一旦协调者发除准备消息，所有参与者马上开始写redo log，涉及的资源都被锁住，如果某个参与者宣告失败，则都做了无用功
  • 一致性风险：在提交阶段和某些参与者连接断开或者出现了网络分区，就会出现部分提交部分回滚的现象

三段提交

为了解决二段提交中存在的单点问题和部分性能问题，后续又发展出了三段提交，将二段提交的准备阶段拆分为两个部分：CanCommit(检验参与者都没有宕机能不能ping通，业务是否能执行完成，但不记录日志)、PreCommit(记录redo log和undo log)，另外三端提交如果超时没有收到协调者发送的提交消息默认执行的是提交

三段提交在事务正确执行的情况下和二段提交性能相似，但回滚时性能有所提升

2.2 CAP理论、BASE理论

从效果看，XA出发点追求的是强一致，实际效果是在节点网络恢复不了，或者某节点宕机且不可恢复时，保证不了一致性；进入提交阶段前，是需要得到所有节点的可提交回复后才可进行，只要有一个节点有性能问题，就会导致整个分布式事务出现性能问题。所以只在单服务多数据源的场景下有应用，分布式环境基于CAP理论和延伸的BASE理论有自己的解决方案。

CAP理论

在一个分布式环境下，设计共享数据问题时，以下三个特性最多只能同时满足其中两个：

  • 一致性（Consistency）：代表数据在任何时刻、任何分布式节点中所看到的都是符合预期的
  • 可用性（Availability）：代表系统不间断地提供服务的能力，理解可用性要先理解与其密切相关两个指标：可靠性和可维护性。可靠性使用平均无故障时间（MTBF）来度量；可维护性使用平均可修复时间（MTTR）来度量。可用性衡量系统可以正常使用的时间与总时间之比，其表征为：A=MTBF/（MTBF+MTTR），即可用性是由可靠性和可维护性计算得出的比例值
  • 分区容忍性（Partition Tolerance）：代表分布式环境中部分节点因网络原因而彼此失联后，即与其他节点形成“网络分区”时，系统仍能正确地提供服务的能力

BASE理论

BASE 分别是基本可用性（Basically Available）、软状态（Soft State）和最终一致性（Eventually Consistent）的缩写

2.3 TCC

TCC是“Try-Confirm-Cancel”三个单词的缩写，是一种业务侵入式较强的事务方案，要求业务处理过程必须拆分为“预留业务资源”和“确认/释放消费资源”两个子过程

  • Try：尝试执行阶段，完成所有业务可执行性的检查（保障一致性），并且预留好全部需用到的业务资源（保障隔离性）。
  • Confirm：确认执行阶段，不进行任何业务检查，直接使用 Try 阶段准备的资源来完成业务处理。Confirm 阶段可能会重复执行，因此本阶段所执行的操作需要具备幂等性。
  • Cancel：取消执行阶段，释放 Try 阶段预留的业务资源。Cancel 阶段可能会重复执行，也需要满足幂等性。

优点：几乎不会涉及到锁和资源的争用，具有很高的性能潜力。

缺点：Try，Confirm，Cancel功能需要自己编码实现，业务侵入性较强，一般会结合框架使用，如：seata，来减轻一些编码工作量；遇到需要调第三方系统难以保证一致性，因为第三方系统不能随心所欲定义冻结、解冻、扣除操作，这时候就可以考虑另一种事务模式：SAGA

2.4 SAGA

适用于业务流程长、有第三方系统参与且需要保证事务最终一致性的业务系统。

把大事务拆分成若干个子事务,T1，T2，…，Ti，…，Tn

每一个子事务都对应一个补偿动作，C1，C2，…，Ci，…，Cn

要求子事务和回滚逻辑都具有幂等性、回滚逻辑和子事务不管谁先执行效果都是一样的

当发生事务回滚时，有两种恢复策略

  • 正向恢复 如果某一步事务提交失败，则一直对该事务进行重试，直到成功为止(最大努力交付)，适用于事务最终都要成功的场景
  • 反向恢复 如果某一步事务提交失败，则对该笔及之前的事务执行定义的回归逻辑(失败则一直重试)

同时，saga也要求记录日志(saga log)来保证系统恢复后可以追踪到子事务的执行情况，比如执行都到哪一步或者补偿到哪一步了。

优点：一阶段提交本地数据库事务，高性能；补偿服务易于理解，易于实现。

缺点：Saga无法保证隔离性，需要额外加锁保证(根据业务场景选择全局写锁还是全局读锁)。对事务和回滚逻辑要满足 幂等性 和 交换律，部分场景下难以实现

2.5 AT

AT 事务是参照了 XA 两段提交协议来实现的，但解决了 XA 2PC 在准备阶段需要等待所有数据源返回成功后，协调者才能发出commit命令而导致的木桶效应。

AT 事务会在收到prepare消息后直接提交本地事务，并记录sql修改前后的数据（和saga一样都是直接提交，但saga通过回滚逻辑回滚，AT通过数据快照回滚），在不加全局锁的情况都有可能在本地事务提交后，数据回滚前发生了数据变化导致 脏写

2.6 GTS

GTS 增加了一个 全局锁 的概念，一般这个锁是写锁，只保证写隔离，因为阻塞读操作的话对性能影响太大

三、透明多级分流

在用户使用信息系统的过程中，请求从浏览器出发，在域名服务器的指引下找到系统的入口，经过网关、负载均衡器、缓存、服务集群等一系列设施，最后触及到末端存储于数据库服务器中的信息，然后逐级返回到用户的浏览器之中。这其中要经过很多技术部件。作为系统的设计者，我们应该意识到不同的设施、部件在系统中有各自不同的价值

  • 有一些部件位于客户端或网络的边缘，能够迅速响应用户的请求，避免给后方的 I/O 与 CPU 带来压力，典型如本地缓存、内容分发网络、反向代理等
  • 有一些部件的处理能力能够线性拓展，易于伸缩，可以使用较小的代价堆叠机器来获得与用户数量相匹配的并发性能，应尽量作为业务逻辑的主要载体，典型如集群中能够自动扩缩的服务节点
  • 有一些部件稳定服务对系统运行有全局性的影响，要时刻保持着容错备份，维护着高可用性，典型如服务注册中心、配置中心
  • 有一些设施是天生的单点部件，只能依靠升级机器本身的网络、存储和运算性能来提升处理能力，如位于系统入口的路由、网关或者负载均衡器（它们都可以做集群，但一次网络请求中无可避免至少有一个是单点的部件）、位于请求调用链末端的传统关系数据库等，都是典型的容易形成单点部件

对系统进行流量规划时，我们应该充分理解这些部件的价值差异，有两条简单、普适的原则能指导我们进行设计：

  • 第一条原则是尽可能减少单点部件，如果某些单点是无可避免的，则应尽最大限度减少到达单点部件的流量。在系统中往往会有多个部件能够处理、响应用户请求，譬如要获取一张存储在数据库的用户头像图片，浏览器缓存、内容分发网络、反向代理、Web 服务器、文件服务器、数据库都可能提供这张图片。恰如其分地引导请求分流至最合适的组件中，避免绝大多数流量汇集到单点部件（如数据库），同时依然能够在绝大多数时候保证处理结果的准确性，使单点系统在出现故障时自动而迅速地实施补救措施，这便是系统架构中多级分流的意义。
  • 另一条更关键的原则是奥卡姆剃刀原则。作为一名架构设计者，你应对多级分流的手段有全面的理解与充分的准备，同时清晰地意识到这些设施并不是越多越好。在实际构建系统时，你应当在有明确需求、真正必要的时候再去考虑部署它们。不是每一个系统都要追求高并发、高可用的，根据系统的用户量、峰值流量和团队本身的技术与运维能力来考虑如何部署这些设施才是合理的做法，在能满足需求的前提下，最简单的系统就是最好的系统。

资源预下载，资源异步加载，资源压缩，dns加速，静态资源内容分发，多机房服务器就近访问，购买多条网络链路，防火墙过滤非法请求，F5硬件层面负载，反向代理软件层面负载均衡，API网关，多实例部署，缓存加速，一致性Hash分流，DB分库，异步任务队列等等都属于分流的手段。

1. 客户端缓存

HTTP 协议的无状态性决定了它必须依靠客户端缓存来解决网络传输效率上的缺陷

从 HTTP/1.0 到 1.1、再到 2.0 版本的演进中，逐步形成了现在被称为“状态缓存”、“强制缓存”（或简称为“强缓存”）和“协商缓存”这三种 HTTP 缓存机制

状态缓存

指不经过服务器，客户端直接根据缓存信息来判断目标网站的状态。如：301//Moved Permanently（永久重定向）、HSTS机制，用来避免依赖 301/302 跳转 HTTPS 时，可能产生的降级中间人劫持问题

强制缓存

如果一段时间内，服务器响应未来一段时间资源内容和状态一定不会发生改变，则客户端可以一直使用该资源的本地副本

根据约定，在浏览器的地址输入、页面链接跳转、新开窗口、前进和后退中，强制缓存都可以生效，但在用户主动刷新页面时应当自动失效

在http协议中，设置了两类可以实现强制缓存的headers

  • Expires：后面跟缓存失效的时间，但因为这个时间是个时间点，所以受客户端本地时间影响；另外因为功能简单，浏览器、代理服务器、cdn不知道哪些该被自己缓存，比如用户敏感信息可以被浏览器缓存，但不应该被中间服务器缓存
  • Cache-Control：http1.1中定义，如果定义上和Expires上有冲突则Cache-Control优先级更高。
    • max-age 和 s-maxage：以秒为单位的缓存有效时长，s表示在允许被中间服务器缓存时，共享缓存的有效时长
    • public 和 private：public表示可以被cdn、代理服务器缓存，private则只被浏览器缓存
    • no-cache 和 no-store：no-cache 表明该资源不应该被缓存、no-store禁止浏览器、cdn等缓存
    • no-transform：禁止资源被转换(压缩等)
    • min-fresh 和 only-if-cached：是客户端请求的header，min-fresh用于建议服务器返回一个不少于该时间的缓存资源，only-if-cached 表示客户端要求不必给它发送资源的具体内容，此时客户端就仅能使用事先缓存的资源来进行响应，若缓存不能命中，就直接返回 503/Service Unavailable 错误
    • must-revalidate 和 proxy-revalidate：must-revalidate 表示在资源过期后，一定需要从服务器中进行获取，即超过了 max-age 的时间后，就等同于 no-cache 的行为，proxy-revalidate 用于提示代理、CDN 等设备资源过期后的缓存行为，除对象不同外，语义与 must-revalidate 完全一致

协商缓存

强制缓存是基于时效性的，而协商缓存是基于缓存内容变化检测的，相比强制缓存有更好的时效性但性能略差

  • Last-Modified 和 If-Modified-Since

    Last-Modified告诉客户端这个资源的最后修改时间，而对于带有这个 Header 的资源，当客户端需要再次请求时，会通过 If-Modified-Since，把之前收到的资源最后修改时间再发送回服务端

    如果此时，服务端发现资源在该时间后没有被修改过，就只要返回一个 304/Not Modified 的响应即可，无需附带消息体，从而达到了节省流量的目的

    而如果此时，服务端发现资源在该时间之后有变动，就会返回 200/OK 的完整响应，在消息体中包含最新的资源

  • Etag 和 If-None-Match

    Etag 是服务器的响应 Header，用于告诉客户端这个资源的唯一标识，对于带有这个 Header 的资源，当客户端需要再次请求时，就会通过 If-None-Match，把之前收到的资源唯一标识发送回服务端

    如果此时，服务端计算后发现资源的唯一标识与上传回来的一致，就说明资源没有被修改过，同样也只需要返回一个 304/Not Modified 的响应即可，无需附带消息体，达到节省流量的目的

    而如果此时，服务端发现资源的唯一标识有变动，也一样会返回 200/OK 的完整响应，在消息体中包含最新的资源。

2. 域名解析

DNS 也许是全世界最大、使用最频繁的信息查询系统，如果没有适当的分流机制，DNS 将会成为整个网络的瓶颈。

DNS 的作用是把便于人类理解的域名地址，转换为便于计算机处理的 IP 地址

DNS的分类

  1. 根dns

    记录着顶级域名和对应的托管商，比如：.com由 Verisign 托管，.apple由苹果公司托管等。由于早期的 DNS 查询结果是一个512字节的 UDP 数据包。这个包最多可以容纳13个服务器的地址，因此就规定全世界有13个根域名服务器，这13台根域名服务器由12个组织独立运营，每家公司为了保证根域名服务器的可用性，会部署多个节点：https://root-servers.org/，一般本机都会保存一份根域名服务器的 IP 地址的缓存

  2. 权威dns

    是经过上一级授权对域名进行解析的服务器，同时它可以把解析授权转授给其他人,我们解析域名的结果都源自权威dns

  3. 递归dns

    负责缓存dns查询结果，避免用户请求重复向上查询。我们平时使用最多的就是这类dns，他对公众开放服务，一般由网络运营商提供服务，比如谷歌的8.8.8.8和8.8.4.4以及114的114.114.114.114和114.114.115.115都属于这一类DNS

  4. 转发dns

    它也具备缓存功能，但这个查询结果不是按标准的 域名解析 过程得到的，而是直接把递归DNS的结果转发给用户。他主要使用在没有直接的互联网连接，但可以连接到一个递归DNS那里，这时使用转发DNS就比较合适，我们路由器里面的dns就是这一类

DNS的完整流程

以www.baidu.com.cn举例，以下是dns解析的完整步骤：

  1. DNS 会把域名还原为www.baidu.com.cn. 最后多了一个点“.”，表示“.root”的意思

  2. 客户端先检查本地递归dns缓存，确认其是否存活(ttl存活时间)，包括后续的每一级dns查询都是有类似的缓存查询的

    检查的顺序为：www.baidu.com.cn的权威服务器 -> baidu.com.cn的权威服务器 -> com.cn的权威服务器 -> cn的权威服务器 -> 根域名服务器

  3. 如果都没有找到会请求根域名服务器拿到cn的权威服务器，然后请求cn的权威服务器得到com.cn的权威服务器，以此类推

域名解析优化

DNS 系统多级分流的设计使得 DNS 系统能够经受住全球网络流量不间断的冲击，但也并非全无缺点，典型的问题是响应速度。

一种方式是根据访问者所处的区域、不同服务商(电信、移动、联通)等因素来确定返回最适合的访问路线。

也有一种前端 DNS预取的方式，如果网站后续要使用来自于其他域的资源，那就在网页加载时生成一个 link 请求，促使浏览器提前对该域名进行预解释

DNS的多级体系还带来一个风险，就是每一级调用的时候都有可能被中间人劫持，这个风险一般来自于递归dns相关链路，甚至不少地区的运营商自己就会主动进行劫持，专门返回一个错的 IP，代理用户请求，以便给特定类型的资源（主要是 HTML）注入广告

为此，最近几年出现了另一种新的 DNS 工作模式：HTTPDNS（也称为 DNS over HTTPS，DoH）。它将原本的 DNS 解析服务开放为一个基于 HTTPS 协议的查询服务，替代基于 UDP 传输协议的 DNS 域名解析，通过程序代替操作系统直接从权威 DNS 或者可靠的 Local DNS 获取解析数据，从而绕过传统 Local DNS

3. 内容分发网络

一个运作良好的内容分发网络（CDN），能为互联网系统解决跨运营商、跨地域物理距离所导致的时延问题，能为网站流量带宽起到分流、减负的作用

内容分发网络的工作过程，主要涉及路由解析、内容分发、负载均衡和所能支持的 CDN 应用内容四个方面

路由解析

内容分发网络将用户请求路由到它的资源服务器上就是依靠 DNS 服务器来实现的

以上是有了dns解析后dns返回的地址信息，由此我们可以了解到cdn路由解析的工作流程

  1. 架设好我们的服务器后，将服务器的ip地址在cdn服务商上注册为“源站”，注册后会得到一个CNAME，就是上面的 icyfenxi.cn.cdn.dnsv1.com，然后需要将得到的CNAME在之前购买域名的DNS服务商那记录一条CNAME记录
  2. 用户请求后，会发现本地递归dns没有缓存，然后请求域名权威dns，获取之前记录的CNAME记录:icyfenxi.cn.cdn.dnsv1.com
  3. 递归dns拿到CNAME后，再次请求CDN服务商所架设的权威服务器，这个 DNS 服务会根据一定的均衡策略和参数，比如拓扑结构、容量、时延等等，在全国各地能提供服务的 CDN 缓存节点中挑选一个最适合的，把它的 IP 替换成源站的 IP 地址，然后返回给本地 DNS
  4. 本地dns将ip返回给浏览器后，浏览器会直接访问这个ip，但此时该ip的dns节点上可能还没有缓存过源站的资源(需要执行内容分发)
  5. 经过内容分发后的 CDN 节点，就有能力代替源站向用户提供所请求的资源了

内容分发

CDN通过内容分发将资源副本缓存在客户端请求的CDN缓存节点上

目前主要有两种内容分发的方式：

  1. 主动分发

    由源站发起，将内容从源站或者其他资源库推送到用户边缘的各个CDN缓存节点上

    需要注意的是，因为主动分发的方式需要需要源站和CDN服务双方提供的程序API接口层面的配合，所以它对源站并不透明，只对用户透明

    这种方式一般适用于网站要预载大量资源的场景，如双十一活动。就会提前把活动用到的资源推送到cdn缓存节点，甚至是手机的app存储空间或者是浏览器的localStorage上

  2. 被动回源

    由用户访问触发，如果CDN缓存节点发现自己没有该资源，就会实时从源站中获取，所以首次访问通常会比较慢，不适合应用于数据量较大的资源，但是被动回源的方式可以做到完全的双向透明，不需要源站做任何配合，所以像阿里云、腾讯云基本都是采用的这种方式

CDN缓存节点资源管理的方式一般是 超时被动失效与手工主动失效相结合，缓存资源会给定生存期，超过了声明周期下次请求时会重新被动回源一次。而手动失效是指，CDN服务商一般会给源站提供失效缓存的接口，在网站更新资源时，由持续集成的流水线自动调用该接口来实现缓存更新

内容分发的应用场景

  • 加速静态资源
  • 安全防御：CDN可以看做源站的堡垒机，源站只对CDN提供服务，然后由CDN来服务外界的其他用户，可以很好的避免如DDos攻击等问题
  • 协议升级：源站可以是http对接CDN节点，但CDN节点提供https，还比如http1.x->http/2|http/3、ipv4->ipv6
  • 状态缓存：源站的 301/302 转向就可以缓存起来，让客户端直接跳转；还可以通过 CDN 开启HSTS、通过 CDN 进行OCSP 装订，来加速 SSL 证书访问
  • 修改资源：可以对源站未压缩的资源自动压缩，并修改 Content-Encoding，以节省用户的网络带宽消耗，可以自动启用客户端缓存，可以加跨域；
  • 访问控制：IP黑/白名单，根据 IP 的访问流量来实现 QoS 控制、根据 HTTP 的 Referer 来实现防盗链
